# P8-0 前端产品化范围定义与路线收敛

---

## 1. 当前背景

P7-I 已完成真实 MiniMax 主链路验证与修复收口，P7-J0 已完成并发架构边界归纳。当前后台核心能力基本稳定，当前支持基础并发和小规模多浏览器访问，不承诺高并发 SaaS 能力，声音克隆/声音设计继续暂缓。

当前需要进入 P8：从"测试面板"转向"产品工作台"。

---

## 2. P8 的核心目标

P8 的目标不是继续扩后端能力，而是把已有能力包装成一个用户能理解、能连续使用、能完成任务的产品界面。

当前已有能力：

- 同步 T2A
- 异步 T2A
- WebSocket 流式
- 批量长文本
- 批量剧本
- 音色试听
- 字幕生成
- 音频播放 / 下载
- 历史记录
- Admin 统计
- Resource Guard 友好提示

P8 要做的是：把这些能力从"接口测试入口"整理成"音频创作工作台"。

## 4. 产品定位

建议定位为：**AI 音频创作工作台**

或者：**短视频旁白与音频剧本生成工作台**

产品价值不是"调用 MiniMax API"，而是：把文本、文案、剧本转成可管理、可试听、可导出的音频资产。

核心价值：

- 音频生成流程封装
- 音色选择和试听
- 长文本分段生成
- 多角色剧本生成
- 字幕生成
- 音频资产管理
- 历史记录和下载
- 本地使用 / 小规模使用成本低

---

## 5. P8 第一版产品形态

**当前建议优先做：本地 Web App / 单用户 Web 工作台**

暂不优先做：

- 完整桌面 App
- 正式 SaaS
- BYOK Web 版
- 开放 API 平台
- 多用户登录系统

**原因**：本地 Web App 可以最大程度复用当前 FastAPI + static 前端结构，成本最低，最适合从测试面板过渡到产品工作台。

产品形态阶段规划：

| 阶段 | 产品形态 | 说明 |
|---|---|---|
| P8 | 本地 Web App / 单用户工作台 | 优先做产品体验 |
| P9 | 本地 App 打包评估 | Electron / Tauri / pywebview |
| 后续 | Web SaaS / BYOK | 等产品闭环验证后再做 |
| 后续 | 开放 API 平台 | 暂不考虑 |

---

## 6. P8 第一版主流程

```
进入工作台
↓
选择创作场景
↓
输入文本 / 剧本
↓
选择音色 / 参数
↓
试听或生成
↓
显示任务卡片
↓
播放音频
↓
查看字幕
↓
下载音频 / 字幕
↓
历史记录中可找回
```

第一版至少支持三条主线：

1. 单段旁白生成
2. 长文本分段生成
3. 多角色剧本生成

## 7. 第一批产品化能力

| 能力 | 是否进入 P8 | 说明 |
|---|---:|---|
| 同步 T2A | 是 | 用于短文本试听 / 快速生成 |
| 异步 T2A | 是 | 用于较长文本生成 |
| WebSocket 流式 | 是 | 用于实时体验，但入口要克制 |
| 批量长文本 | 是 | 核心生产能力 |
| 批量剧本 | 是 | 多角色剧本核心能力 |
| 字幕生成 | 是 | 与音频结果绑定展示 |
| 音色试听 | 是 | 产品化重点 |
| 历史记录 | 是 | 资产管理核心 |
| 音频下载 | 是 | 必须完善体验 |
| Resource Guard 提示 | 是 | 保留友好提示 |
| Admin 统计 | 内部使用 | 不进入普通用户主界面 |

---

## 8. 暂缓能力

| 能力 | 原因 |
|---|---|
| 声音克隆 | 高成本，需样本质量控制，暂不进入第一版 |
| 声音设计 | 高成本，效果主观，暂不进入第一版 |
| BYOK | 需要用户/session/credential 体系，后续再评估 |
| 多用户登录 | 当前阶段不做 SaaS |
| 开放 API | 复杂度高，暂不做 |
| 队列 / worker | 当前小规模使用暂不需要 |
| Redis / PostgreSQL | 当前不进入 P8 |
| HTTP 流式端点 | 产品/API 口径待决策，暂不作为 P8 主任务 |

## 9. 前端现状问题

当前前端是 `app/static/index.html` 单文件测试面板。

问题：

- 功能入口多，但不像产品
- 页面信息密度高
- 用户不知道先做什么
- 音色选择体验弱
- 任务结果展示偏测试
- 历史记录和下载体验不够产品化
- 移动端适配不足
- 成本估算和 profiles 加载可优化
- 刷新后任务恢复不足

---

## 10. P8 前端结构建议

建议前端按区域组织，而不是继续堆测试入口。

推荐结构：

```
顶部导航
├── 创作工作台
├── 音色工作台
├── 批量任务
├── 历史记录
├── 设置
└── Admin 入口（内部）
```

第一版页面结构可以是单页多 tab：

1. 单段生成
2. 长文本生成
3. 剧本生成
4. 音色试听
5. 历史记录
6. 设置

---

## 11. 任务卡片设计

P8 必须引入任务卡片概念。

任务卡片应包含：

- 任务类型
- job_id / batch_id
- 当前状态
- 已等待时间
- 文本预览
- 使用音色
- 音频播放器
- 字幕查看
- 下载按钮
- 手动刷新
- 错误提示
- 重新生成入口

异步任务和批量任务不应该让用户一直盯着按钮等待，而应该通过任务卡片展示进度。

---

## 12. localStorage 最近任务恢复

P8 建议做轻量任务恢复，不做完整幂等。

保存内容：

- last_async_job_id
- last_batch_id
- task_type
- created_at
- text_preview
- status

刷新后提示："检测到上次未完成任务，是否继续查看？"

提供按钮：继续查看 / 忽略 / 清除记录

**说明**：这不是完整幂等系统，只解决单浏览器刷新后的任务追踪问题。

---

## 13. 轻量防误点策略

P8 做轻量防误点：

- 当前按钮点击后 loading
- 当前请求返回前禁用当前按钮
- 不锁整个页面
- 异步 submit 返回 job_id 后恢复按钮
- 批量 submit 返回 batch_id 后恢复按钮
- 流式生成期间限制同页面只有一个流式连接

不做：

- 后端完整幂等表
- 多用户重复请求去重
- 跨浏览器幂等

---

## 14. 请求优化

P8 可以纳入两个 P3 优化：

**cost estimate debounce**：输入停止 500ms-800ms 后再请求。

**profiles 缓存**：页面加载后缓存 profiles，多个模块复用，减少重复请求。

## 15. 设置页规划

如果后续要走本地 App 路线，设置页需要预留：

- MiniMax API Key 配置
- 默认模型
- 默认音色
- 默认导出目录
- 存储目录
- 清理缓存
- 测试连接

P8-1 不一定马上实现 API Key 设置页，可以先作为规划记录。

---

## 16. P8 不做的事情

P8 不做：

- 完整登录系统
- 多租户
- SaaS 计费
- 开放 API
- BYOK credential 管理
- Redis / 队列 / worker
- PostgreSQL 迁移
- 声音克隆产品化
- 声音设计产品化
- 大规模并发压测

|---|---|

## 18. P8-1 建议优先做什么

P8-1 不建议直接换 React / Vue。

建议先在当前 `app/static/index.html` 上做结构整理：

- 顶部导航
- 主工作台区域
- 右侧结果区域
- 场景 tab
- 统一任务卡片样式
- 保留现有 API 调用逻辑
- 不大改后端

**目标**：先让页面从测试面板变成可理解的工作台。

---

## 19. 阶段结论

P8 应聚焦产品化体验，不应继续扩张后端架构。

第一版产品形态建议先做本地 Web App / 单用户音频创作工作台。

核心目标是把已有音频能力组织成清晰的创作流程、音色体验、任务结果和历史资产管理。
