目录架构层当前职责与 Resource Guard 的关系
app/api 接入层 接收 HTTP / WebSocket 请求，转换为 schema 通常不放复杂限流逻辑，只透传 request 信息
app/services 业务编排层 组织校验、生成、导入、克隆、批量等业务流程 Resource Guard 第一版主要接入点
app/domain 领域协议层 RenderPlan、请求/响应 schema、枚举 后续定义 OperationPolicy、ResourcePolicy、RenderPlan v2
app/providers 底层适配层 将统一 RenderPlan 翻译成 MiniMax/mock 请求 不建议把 Resource Guard 写进 Adapter；Adapter 只负责调用底层
app/models / repositories 数据层 保存 ProviderVoice、Binding、Job、Asset、CallLog 后续 Resource Guard 如果要持久化队列或预算，会涉及这里
app/static 前端层 当前测试面板，已有 guardedJsonFetch 前端可防重复点击，但后端 Resource Guard 才是安全边界

4. 核心对象关系图：为什么不是直接传 voice_id

Voice Lab 的核心抽象是：业务上使用 VoiceProfile，底层真实可调用的是 ProviderVoice，中间通过 VoiceBinding 连接。RenderPlan 则是一张执行说明书。

这些链路都会触发真实 Provider 调用。Resource Guard 不能只写在单个接口，而应该作为所有真实 Provider 调用前的统一资源闸门。

6. 待开发模块的意义：为什么先理解再开发

下面不是功能清单，而是架构模块清单。每个模块都解决一个系统性问题。

下一步候选

ResourceGuardService

解决“现在能不能执行”。按 provider + model + operation + api_key_id 控制并发，避免云端费用和限流失控。

架构收敛

OperationPolicy

集中声明每个 operation 的风险、是否需要 CostGuard、ResourceGuard、ProviderVoice、usage 回填。

多 Provider 基础

ProviderCapabilities

让每个 Provider 自己声明支持什么，避免前端和 service 到处写 provider == minimax。

多模型基础

RenderPlan v2

让生成计划能表达 source_type、metadata、voice_prompt、reference_audio_path、provider_options。

音色资产标准

ProviderVoiceMetadataSpec

规范 metadata_json，支持 remote_voice_id、stateless_design、stateless_clone、local_reference 等声音来源。

成本系统

BudgetGuard

从 confirm_cost 防误触升级为 estimate_id、预算预占、实际 usage 结算、用户/API Key 限额。

可观测性

ProviderCallContext

统一记录 provider、operation、job_id、trace_id、duration、usage、error，避免 usage 回填散落。

产品化

创作工作台

把测试面板改成“选择场景 → 选择声音 → 输入文案 → 成本提示 → 生成试听”的产品主流程。

架构师结论： 当前项目的首要问题不是“功能少”，而是“横切能力开始复杂”。Resource Guard 开发前，应该先明确 operation、provider、model、api_key_id 四个维度，否则很容易又变成每个 service 各写一套 semaphore。
