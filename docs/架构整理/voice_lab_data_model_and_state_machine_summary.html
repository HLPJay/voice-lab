Voice Lab 核心架构汇总

数据模型与状态机：项目可扩展性的核心骨架

这份文档只聚焦两个方向：第一，当前 Voice Lab 的数据模型如何组织，以及它为什么具备扩展能力； 第二，状态机如何承载同步、异步、流式、批量等不同任务下发，并让任务过程可追踪、可恢复、可治理。

一、总览：为什么数据模型和状态机是架构核心

数据模型决定系统“能表达什么”，状态机决定这些对象“如何变化”。

Voice Lab 不是一个简单的 MiniMax API Demo，而是一个逐步演进中的声音生产工作台。 它需要管理声音人设、Provider 音色、绑定策略、生成任务、音频资产、批量分段和调用日志。 如果数据模型没有抽象好，后续每接入一个 Provider 都会改业务；如果状态机没有设计好，异步、流式、批量任务就会出现长期 processing、失败不可追踪、成功无资产等问题。

数据模型回答的问题
  • 系统里有哪些核心对象？
  • 用户看到的“声音”和 Provider 的 voice_id 如何解耦？
  • 生成任务和生成结果如何关联？
  • 未来如何接入更多 Provider？
状态机回答的问题
  • 任务从创建到完成经历哪些状态？
  • 同步、异步、流式、批量任务如何统一管理？
  • 失败如何落库？成功如何证明？
  • 批量部分失败如何表达？
Voice Lab 核心骨架：

数据模型：
  VoiceProfile / ProviderVoice / VoiceBinding / RenderPlan / VoiceJob / Asset

状态机：
  JobType + JobStatus + BatchStatus + 资产生命周期状态

控制层：
  Binding Validation / Cost Guard / Resource Guard

三者关系：
  数据模型定义对象
    ↓
  状态机定义对象变化
    ↓
  控制层决定哪些变化或调用允许发生
核心结论：当前项目的数据模型方向是正确的，已经具备声音中台雏形；状态机基本可用，但后续需要围绕 operation、异步超时、批量汇总等方向继续精细化。

二、数据模型汇总：当前系统如何表达声音业务

当前数据模型的设计思路是：不要让业务层直接依赖 MiniMax 原始 voice_id，而是通过一层自己的业务抽象，把“用户想要的声音”和“Provider 真实音色”隔离开。

2.1 数据模型分层

声音抽象层：VoiceProfile
表达用户理解的声音人设，例如“奶娘”“深夜程序员”“旁白”。它描述风格，不直接绑定某个厂商的 voice_id。
↓
Provider 资产层：ProviderVoice
表达 Provider 侧真实音色资产，例如 MiniMax 的某个 provider_voice_id。它管理厂商音色、状态和 metadata。
↓
绑定决策层：VoiceBinding
把 VoiceProfile 和 ProviderVoice 连接起来，决定某个人设使用哪个 provider、model、voice_id、参数和优先级。
↓
标准协议层：RenderPlan
统一描述一次生成计划，包括文本、provider、model、provider_voice_id、voice_params、audio_params、subtitle 等。
↓
任务与结果层：VoiceJob → AudioAsset / SubtitleAsset
VoiceJob 记录生成过程和状态；AudioAsset / SubtitleAsset 记录生成结果和文件资产。

2.2 核心模型清单

模型 一句话理解 主要字段 / 信息 架构价值
VoiceProfile 声音人设 name、description、gender_style、age_style、tone_style、emotion_style、scene_tags_json、is_active 把“用户想要的声音风格”抽象出来，避免直接依赖 Provider voice_id。
ProviderVoice Provider 音色资产 provider、provider_voice_id、voice_type、name、language、gender、status、metadata_json、synced_at 统一管理多个 Provider 的真实音色，并通过 provider + provider_voice_id 避免冲突。
VoiceBinding 人设到音色的绑定策略 profile_id、provider、model、provider_voice_id、params_json、priority、status 让人设和真实 Provider 音色解耦，支持切换、废弃、优先级和未来 fallback。
RenderPlan 标准生成计划 text、processed_text、profile_id、provider、model、provider_voice_id、voice_params、audio_params、subtitle、output_format 把业务请求转成统一协议，Provider Adapter 只负责把 RenderPlan 转换成厂商请求。
VoiceJob 一次生成任务 job_type、status、provider、model、profile_id、binding_id、input_text、processed_text、render_plan_json、response_json、error_message 统一承载同步、异步、流式、批量等任务过程，方便历史、状态、错误和统计。
AudioAsset 音频结果资产 job_id、provider、model、file_path、file_url、format、duration_ms、sample_rate、usage_characters、metadata_json 把生成结果资产化，支撑下载、复用、历史记录、缓存和素材库。
SubtitleAsset 字幕结果资产 job_id、audio_asset_id、subtitle_type、file_path、srt_path、timeline_json 让字幕与音频结果可追踪、可合并、可导出。
ProviderCallLog 真实 Provider 调用日志 provider、api_path、status_code、duration_ms、usage_characters、provider_trace_id、error_message 支撑调用审计、成本统计、失败排查和 Provider 质量评估。

2.3 数据模型的扩展机制

业务和厂商解耦

用户选择 VoiceProfile，系统通过 VoiceBinding 找到 ProviderVoice。用户不直接依赖 MiniMax voice_id。

Provider 多态

ProviderVoice 用 provider + provider_voice_id 表达真实音色，未来可以同时支持 MiniMax、MiMo、ElevenLabs、本地模型。

绑定策略可变

VoiceBinding 包含 model、params_json、priority、status，可以支持优先级、废弃、切换、fallback。

标准生成协议

RenderPlan 是内部标准协议，业务不直接拼厂商请求。新增 Provider 只需要新增 Adapter。

任务与结果分离

VoiceJob 记录过程，AudioAsset 记录结果。过程失败不会污染结果资产，结果资产可以复用。

JSON 扩展空间

params_json、metadata_json、response_json 提供早期扩展性，但未来高频统计字段应结构化。

2.4 一个完整生成请求中的数据模型协作

用户输入：
  profile_id = “奶娘”
  text = “今天的故事开始了”

系统内部：
  1. 查 VoiceProfile
  2. 查 VoiceBinding(profile_id, provider)
  3. 查 ProviderVoice(provider, provider_voice_id, status=available)
  4. 构造 RenderPlan
  5. 创建 VoiceJob
  6. 调用 Provider Adapter
  7. 保存 AudioAsset / SubtitleAsset
  8. 更新 VoiceJob.status = success / failed
架构意义：用户只关心“我要什么声音”，系统内部通过数据模型完成“这个声音由哪个 Provider、哪个模型、哪个 voice_id、哪些参数来实现”的决策。

三、状态机汇总：不同任务如何统一下发与追踪

Voice Lab 的任务不是单一类型。同步生成、异步生成、流式生成、批量任务的执行方式不同， 但它们都需要统一的任务生命周期。当前项目通过 JobType + JobStatus + BatchStatus 来完成这一点。

3.1 当前状态枚举

枚举 状态值 用途
JobStatus pending / running / processing / success / failed 描述单个 VoiceJob 的生命周期。
JobType sync_render / async_render / stream_render / batch_render 描述任务执行类型。
BatchStatus pending / running / success / partial / failed 描述批量任务的整体汇总状态。
BindingStatus available / deprecated 描述绑定是否可用于新生成。
ProviderVoiceStatus available / deprecated 描述 Provider 音色是否可用于新生成。

3.2 通用 VoiceJob 状态机

VoiceJob 通用生命周期：

pending
  ↓
running / processing
  ↓
success / failed
pending

任务已创建，但尚未真正开始执行。

running

本地进程正在执行任务，例如同步生成、流式连接期间。

processing

任务已提交，外部 Provider 或后台流程仍在处理中，例如异步 T2A。

success / failed

任务进入明确终态。success 应该有结果资产，failed 应该有错误信息。

3.3 不同 JobType 的状态流转

任务类型 状态流转 说明 关键风险
sync_render pending → running → success / failed 同步生成在一次请求内完成。 Provider 调用失败后必须 failed；success 前必须保存 AudioAsset。
async_render pending → processing → success / failed 异步任务提交后由 Provider 后台处理，前端轮询推进状态。 前端停止轮询、Provider expired、file_url 缺失可能导致长期 processing。
stream_render pending → running → success / failed WebSocket 连接期间持续输出音频 chunk。 连接异常断开时必须释放资源，并进入明确状态。
batch_render pending → running → success / partial / failed 批量任务由多个 segment 汇总结果。 部分段落失败时不能丢掉成功段；合并失败要有明确规则。

3.4 Batch 状态机

BatchJob：
pending
  ↓
running
  ↓
success / partial / failed

BatchSegment：
pending
  ↓
running
  ↓
success / failed / skipped
Batch 终态 含义 适用场景
success 所有 segment 成功，音频和字幕合并成功。 完整批量任务成功。
partial 部分 segment 成功，部分失败；或主结果部分可用。 允许保留成功段，后续重试失败段。
failed 整体无法执行，或全部 segment 失败。 初始化失败、绑定失败、Provider 全部失败。
批量任务的关键：BatchJob 的状态应该由 BatchSegment 汇总。成功段要能复用，失败段要能重试，不能因为一个分段失败就抹掉全部成果。

3.5 ProviderVoice / VoiceBinding 生命周期

ProviderVoice：
available
  ↓
deprecated

VoiceBinding：
available
  ↓
deprecated

这里使用软废弃，而不是硬删除。原因是历史任务可能仍然引用旧音色或旧绑定。 deprecated 的意义是：不允许新生成继续使用，但历史数据保留。

关键规则：生成前不能只检查 VoiceBinding.status，还要检查 ProviderVoice.status。Binding 可用不代表 ProviderVoice 一定可用。

3.6 状态机设计原则

  1. 所有任务必须进入明确终态：不能长期停在 running / processing。
  2. success 必须有结果资产：没有 AudioAsset 的 success 是不完整状态。
  3. failed 必须有错误信息：没有 error_message 的 failed 难以排查。
  4. 状态变化必须更新 updated_at：用于判断任务是否卡住。
  5. 异步任务要处理跨请求风险：前端停止轮询、Provider expired、file_url 缺失。
  6. 批量任务要支持 partial：允许部分结果可用，避免全盘失败。
  7. 控制层失败也要进入明确处理：如果 job 已创建，Resource Guard 拒绝时应标记 failed。

四、数据模型与状态机如何协作

数据模型负责承载对象，状态机负责管理这些对象的生命周期。一次真实生成任务中，两者是同时工作的。 

一次生成任务的协作流程：

1. 用户选择 VoiceProfile
2. 系统查 VoiceBinding
3. 系统校验 ProviderVoice.status == available
4. 构造 RenderPlan
5. 创建 VoiceJob(status=pending)
6. 通过 Cost Guard / Resource Guard
7. 调用 Provider Adapter
8. 保存 AudioAsset / SubtitleAsset
9. VoiceJob.status = success

如果失败：
  VoiceJob.status = failed
  VoiceJob.error_message = 错误原因

4.1 为什么任务要记录 RenderPlan

VoiceJob.render_plan_json 的价值是复盘。它可以记录当时到底使用了哪个 provider、model、voice_id、文本、参数和输出格式。 这让后续排查“为什么这次生成效果不一样”成为可能。

4.2 为什么结果要拆成 Asset

如果 VoiceJob 只记录任务状态，而 AudioAsset 记录真实结果文件，系统就可以支持历史下载、结果复用、批量合并、缓存和素材库。 任务失败不会污染资产，资产存在也可以被多个上层流程复用。

4.3 为什么 Binding 和 ProviderVoice 都要有 status

Binding 表示“这个人设是否还允许使用这个绑定策略”，ProviderVoice 表示“这个真实厂商音色是否还可用”。 两者状态不完全等价。ProviderVoice 被删除或废弃后，相关 Binding 也应该废弃，但历史记录仍然保留。

五、当前自检结论：优点、潜在问题与演进方向

5.1 当前架构优点

人设和音色解耦

VoiceProfile / ProviderVoice / VoiceBinding 三层设计，使用户声音选择不直接依赖 Provider voice_id。

Provider 可扩展

ProviderVoice + RenderPlan + Provider Adapter 让未来接入其他 Provider 成为可能。

任务过程可追踪

VoiceJob 记录输入、状态、render_plan_json、response_json、error_message。

结果资产化

AudioAsset / SubtitleAsset 让生成结果可以下载、复用、合并和沉淀为素材。

软废弃机制

available / deprecated 能保留历史关系，避免硬删除破坏历史任务。

批量支持 partial

BatchStatus 有 partial，说明系统已经考虑部分成功的真实业务场景。

5.2 当前潜在问题

问题 影响 当前建议
VoiceJob 缺少 operation 字段 难以区分普通生成、试听、设计、克隆、导入验证等业务动作。 先记录，不急改；等 Resource Guard 接入和统计需求稳定后再考虑。
JobType 粒度偏粗 只有 sync / async / stream / batch，业务语义不够细。 未来可采用 job_type + operation 双维度。
running 和 processing 边界需要文档化 多人开发时可能混用状态。 建议定义：running = 本地执行中，processing = 外部 Provider / 后台处理中。
异步任务可能长期 processing 前端停止轮询或 Provider 边界状态会导致任务不进入终态。 后续单独设计超时、refresh、reconciler。
批量状态汇总规则需要更明确 partial / failed / success 的边界可能在复杂场景下混乱。 建议后续沉淀 Batch 状态机文档。
JSON 字段长期统计能力有限 如果某些字段要统计，只放 JSON 会不方便。 等统计需求明确后再结构化。

5.3 当前是否需要立刻优化

结论：当前不建议立刻大改数据模型和状态机。P6 baseline 已稳定，P7 Resource Guard 正在推进，底层模型大改会扩大风险。更合理的方式是先记录架构债，继续完成 Resource Guard，再根据真实使用和统计需求决定是否升级模型。
