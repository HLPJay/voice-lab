Voice Lab 架构自检与后续优化判断

数据模型、状态机、安全性与传输协议选择

这份文档整理了关于当前项目是否需要优化、哪些安全性问题更值得优先处理，以及自定义结构体 / JSON 协议 / Protobuf 之间如何取舍的讨论结论。它适合作为项目存档、后期复盘和后续开发决策依据。

一、当前是否需要优化：结论先行

当前不建议马上大改数据模型和状态机；更值得优先推进真实调用安全与资源保护。

现在不建议动

不要马上改 VoiceJob、JobType、ProviderVoiceStatus、数据库字段结构化等底层模型。

原因：P6 baseline 已稳定，P7 Resource Guard 正在推进。此时改底层模型会引入迁移、兼容、测试和历史数据问题。

现在应该做

继续推进 P7 Resource Guard：先实现基础模块，再分阶段接入核心路径。

原因：当前真实风险不是“模型不够优雅”，而是多条真实 MiniMax 调用路径还没有统一资源闸门。

可以记录不开发

将数据模型、状态机、控制层的架构债整理到文档，例如 CORE_ARCHITECTURE_MAP.md。

目的：帮助后续分析，不破坏当前稳定状态。

1.1 哪些数据模型问题暂时只记录

潜在问题 为什么确实存在 为什么现在不急着改
VoiceJob 缺少 operation 字段 当前 job_type 只能区分 sync / async / stream / batch，不能区分普通生成、试听、克隆、设计等业务动作。 Resource Guard 还没接入业务链路；现在改会涉及模型迁移和多处 Service 调整。
JobType 粒度偏粗 voice_design、voice_clone、preview 是否应独立成 JobType，当前还不清晰。 可等真实审计和统计需求明确后再扩展。
ProviderVoiceStatus 状态偏少 目前只有 available / deprecated，未来可能需要 missing_remote、invalid、pending_verify 等。 当前 Provider 同步复杂度还不高，暂时够用。
部分 JSON 字段未来统计困难 metadata_json、response_json、params_json 灵活但不利于查询统计。 需要等哪些字段真正进入统计和查询后，再结构化。
当前策略：底层模型大改后置；先把真实调用的资源和成本控制做好。

二、当前优先做 Resource Guard 的原因

当前项目已经具备同步、异步、流式、声音设计、声音克隆、试听、批量生成等多条真实 Provider 调用路径。 这些能力已经足够丰富，继续产品化前最需要的是“真实调用可控”。 

当前更现实的风险：
重复点击
  ↓
多次真实调用
  ↓
费用放大

批量任务并发
  ↓
多个段落同时生成
  ↓
Provider 限流或成本失控

WebSocket 流式长连接
  ↓
长时间占用资源
  ↓
其他请求被拖慢或失控

2.1 Resource Guard 当前路线

阶段 目标 是否改业务链路
P7-B 实现 ResourceGuardService 基础模块 + 单元测试 不接业务
P7-C 接入核心同步与高风险路径，如同步生成、设计、克隆、试听 小范围接入
P7-D 接入流式与异步路径 谨慎接入

三、安全性上还有哪些可优化

安全性要分阶段处理。当前最优先的是成本和真实调用安全，而不是立刻做复杂用户系统。

3.1 第一优先级：真实调用安全

风险 表现 优化方向 当前阶段建议
重复点击扣费 用户连续点击生成，触发多次 MiniMax 调用。 Resource Guard、前端按钮禁用、未来 Request Cache。 优先
高风险操作误触 声音设计、声音克隆、批量任务消耗较高。 Cost Guard confirm_cost。 已做第一版，继续保持
批量任务打爆 Provider 多批量、多 segment 并发导致成本和限流问题。 batch_longtext / batch_script 入口限流，后续 execute 生命周期保护。 P7-E
WebSocket 长连接占用 流式生成连接长时间占用资源。 t2a_stream limit=1，断开必须 release。 P7-D

3.2 第二优先级：接口访问安全

如果项目只在本地自己测试，问题不大；但如果开放给别人试用，就必须考虑访问控制。

  • Admin 接口鉴权。
  • 测试面板访问控制。
  • 真实 Provider 调用接口鉴权。
  • CORS 限制。
  • API Key 不能暴露给前端。
  • 静态页面不要裸露高风险真实调用按钮给未授权用户。
小范围试用前，应至少给 Admin 面板和真实调用接口加简单鉴权；正式产品化前，再做完整用户系统和权限模型。

3.3 第三优先级：上传文件安全

声音克隆涉及音频上传，上传功能天然需要更严格的安全边界。

安全点 要防什么
文件大小限制防止超大文件耗尽磁盘或内存。
文件类型校验防止伪装成音频的恶意文件。
MIME + 扩展名双校验避免只靠文件后缀判断。
文件名清洗防止路径穿越，例如 ../xxx。
临时文件清理避免失败上传长期占用空间。
音频时长限制防止不符合克隆要求或过长音频。

3.4 第四优先级：日志和隐私安全

  • 不在日志中打印 API Key。
  • 用户输入文本必要时截断，不要全量无脑记录。
  • Provider 原始响应要谨慎落日志。
  • error_message 做长度限制。
  • Admin 面板不要泄露密钥、内部绝对路径。
  • 导出日志时避免包含用户隐私。

3.5 第五优先级：下载和文件访问安全

  • 只能下载 storage 目录下的文件。
  • 防止通过 file_path 访问系统任意文件。
  • 下载 URL 不要暴露服务器绝对路径。
  • 历史音频资产未来需要权限校验。
  • 临时文件和长期文件要区分。
安全优化总原则：当前先管真实调用成本和资源；小范围试用前补接口鉴权、上传安全、日志脱敏；正式产品化前再做用户系统、权限模型、预算系统和文件访问权限。

四、传输协议：什么时候自定义结构体，什么时候 Protobuf

当前项目最适合继续使用 Pydantic / SQLModel / JSON 协议，暂时不需要 Protobuf。

4.1 先区分三类协议

类型 典型技术 在 Voice Lab 中的角色
内部结构体协议 Pydantic、dataclass、SQLModel 定义 RenderPlan、请求 schema、数据库模型。
外部 JSON 协议 REST JSON、WebSocket JSON event 前端和后端通信，方便调试。
二进制跨语言协议 Protobuf、gRPC 适合多语言服务、高性能 RPC、稳定 SDK。

4.2 什么时候适合自定义结构体

也就是用 Pydantic / dataclass / SQLModel 来定义标准结构。

适合场景
  • Python 单体项目。
  • 协议还在快速迭代。
  • 业务字段经常调整。
  • 需要可读、可调试。
  • 前后端主要通过 JSON 通信。
  • 还没有多语言 SDK 强需求。
Voice Lab 示例

RenderPlan 就是内部标准结构体：它统一描述 provider、model、provider_voice_id、text、voice_params、audio_params、subtitle、output_format。

Provider Adapter 只需要把 RenderPlan 转成 MiniMax 或未来 Provider 的请求体。

4.3 什么时候适合 JSON 事件协议

WebSocket 流式场景更适合 JSON event，而不是立刻上二进制协议。

{
  "event": "audio_chunk",
  "job_id": "job_xxx",
  "seq": 1,
  "audio": "base64..."
}
  • 浏览器前端可以直接消费。
  • 调试时能直接看消息内容。
  • 事件类型清楚，消息量暂时不大。
  • 不需要引入额外编译和解码工具链。

4.4 什么时候适合 Protobuf

Protobuf 适合更成熟、更高性能、更跨语言的系统阶段。

适合 Protobuf 的条件 说明
多语言服务通信例如 Python 后端 + Go 音频服务 + Rust 网关。
稳定长期协议字段不再频繁变化，需要强版本兼容。
高吞吐 / 低带宽 / 低延迟JSON 编解码或体积成为瓶颈。
需要生成多端 SDK例如 iOS / Android / Web / 服务端 SDK。
微服务 RPC适合 gRPC + Protobuf。
移动端 Native 高频通信节省带宽和流量，增强版本约束。

4.5 什么时候不适合 Protobuf

当前 Voice Lab 不适合马上引入 Protobuf。项目还在快速迭代，字段和状态机仍在调整，前端是网页，JSON 更自然；此时引入 Protobuf 会增加 proto 文件维护、生成代码、版本兼容、前端解码和调试复杂度。

4.6 协议选择建议表

场景 推荐方式 原因
后端内部标准生成计划 Pydantic / dataclass 易读、易改、和 Python 生态兼容。
数据库模型 SQLModel 当前项目已经使用，适合快速开发。
REST API JSON + Pydantic schema 前端友好，调试方便。
WebSocket 流式事件 JSON event protocol 浏览器直接消费，排查方便。
Provider Adapter 输入 RenderPlan 结构体 隔离厂商差异。
多语言微服务 RPC Protobuf / gRPC 强 schema、性能好、跨语言。
移动端高频二进制通信 Protobuf 节省带宽，版本管理强。
音频文件传输 文件 URL / bytes stream 不建议把大音频塞进 Protobuf 控制消息。

五、推荐路线图

根据当前项目阶段，建议分三段推进。

5.1 当前阶段：P7 资源保护优先

  • 继续完成 P7-B ResourceGuardService。
  • P7-C / P7-D / P7-E 分阶段接入业务路径。
  • 不要在这个阶段做数据模型大改。
  • 不要引入 Protobuf。
  • 把架构债记录到文档，等待需求稳定后再处理。

5.2 小范围试用前：补基础安全

  • Admin / 真实调用接口加简单鉴权。
  • CORS 限制。
  • 上传文件安全检查。
  • 日志脱敏和错误截断。
  • 前端高风险操作提示。

5.3 正式产品化前：再做系统性升级

  • 用户系统和权限模型。
  • Budget Guard。
  • Request Cache。
  • 文件访问权限。
  • 数据保留策略。
  • 多进程 / Redis 级 Resource Guard。
  • 协议稳定后，再评估 Protobuf / gRPC。
当前最优路线：
P7 Resource Guard
  ↓
小范围试用安全
  ↓
产品主流程稳定
  ↓
再评估数据模型扩展
  ↓
再评估 Protobuf / 微服务协议

六、最终结论

现在最该优化的是真实调用安全和资源控制；数据模型大改、状态机扩展、Protobuf 协议化都应该后置。

数据模型 当前方向正确，有架构债但不急改。先记录，等统计、审计、产品化需求明确后再改。
安全性 当前最重要的是 Resource Guard、Cost Guard、批量保护、上传安全、日志脱敏。
协议选择 当前继续用 Pydantic / SQLModel / JSON。等多语言、高吞吐、稳定 SDK 需求出现后再考虑 Protobuf。
判断标准：如果一个优化会破坏 P6 baseline、扩大修改范围、引入迁移风险，但短期无法明显降低真实调用风险，那么它就不应该现在做。
