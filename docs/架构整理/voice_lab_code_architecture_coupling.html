Voice Lab 代码架构分层、调用与耦合关系图

这版专门解释“代码架构设计”：项目当前分几层、层之间如何调用、哪些层耦合强、后续扩展 Resource Guard / MiMo / RenderPlan v2 的影响范围是什么。

1. 先看心智模型 2. 分层依赖图 3. 目录职责映射 4. 调用链细节 5. 耦合关系 6. 扩展影响范围

1. 先看心智模型：Voice Lab 不是“一堆接口”，而是三件事

当前项目最合理的理解方式：声音资产管理 + 生成任务编排 + Provider 适配。Service 层是业务编排中心，RenderPlan 是 Service 与 Provider 之间的协议。

一句话：Service 层现在最重，它同时连接 API、数据、Guard、RenderPlan、Provider 和资产保存。后续做 Resource Guard，不应该写进 Provider Adapter，而应该作为 Service 调用 Provider 前的统一资源闸门。

2. 分层依赖图：谁可以调用谁，谁不应该反向依赖

当前项目基本是单向依赖。扩展时最重要的是保持这个方向，避免 Provider 反向依赖 Service，避免 Repository 反向依赖业务逻辑。

4. 真实调用链：同步生成、异步生成、后续 Resource Guard 插入点

Resource Guard 不是写在前端，也不应写进 Provider Adapter。它应插在 Service 调用 Provider 之前，和 Cost Guard 一样作为横切能力。

5. 当前耦合关系：哪些地方强，哪些地方弱

耦合不是坏事。业务编排层天然会比较强耦合。关键是要知道强耦合在哪里，并用横切模块把它收敛。

耦合关系 强度 为什么耦合 当前风险 后续解法
Service ↔ Repository/Model 强 Service 创建 Job、查 Binding、保存资产、更新状态 每条链路都自己处理状态和异常 OperationExecutor / JobContext
Service ↔ Provider 强 Service 通过 get_provider 调用 render/clone/design/stream 没有 ProviderCapabilities，容易 provider 特判 ProviderCapabilities + OperationPolicy
Service ↔ RenderPlan 强 Service 构造计划，Provider 消费计划 RenderPlan 偏 MiniMax，扩展影响大 RenderPlan v2，向后兼容新增字段
前端 ↔ API Schema 中 前端直接拼请求体和读取响应结构 schema 变更会影响 index.html 前端按 capabilities 渲染；API 保持稳定
Provider ↔ 外部模型 强 Adapter 必须理解 MiniMax/MiMo 请求格式 外部 API 差异大 把差异封装在 Adapter，不泄漏到 Service
Repository ↔ Model 中 Repository 操作 SQLModel 模型字段变更影响 repository 保持 repository 简单，不写业务策略
Cost Guard ↔ Service 低到中 Service 只声明 operation 和 confirm_cost 当前设计较好 Resource Guard 按同样模式实现
